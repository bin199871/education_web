# 讲解动画已知问题与修复方向

## 验证方法

以下所有问题均使用此题目验证：

> 在竖直平面内存在水平向右的匀强电场，电场强度大小 E = 4.0 × 10³ N/C。一个带正电的小球质量 m = 0.2 kg，电荷量 q = 1.0 × 10⁻⁴ C，用长度 L = 0.8 m 的不可伸长的绝缘细线悬挂于 O 点。现将小球拉到细线与竖直方向夹角 θ₀ = 60° 的位置，由静止释放。
>
> 求：
> （1）电场力 F电 的大小，合力 F合 的大小和方向
> （2）小球第一次运动到最低点 A 时的速度大小 vA
> （3）电势能变化量 ΔEp电 和重力势能变化量 ΔEp重
> （4）细线拉力的最大值 Tmax

确认注入参数结果：

```json
{"m": 0.2, "q": 0.0001, "E": 4000.0, "L": 0.8, "g": 10.0, "theta_target": 37}
```

物理参数提取正确 ✅，但模板渲染存在以下问题。

## 问题清单

### 问题 1：模板面板数不匹配题目问数

**表现：** 题目有 4 问，模板只显示 3 个解答面板。

**根因：** `electric_pendulum.html` 的 `Panel.stepMap` 硬编码了 3 组：

```js
stepMap: {
  sol_q1: {prefix:'q1', count:8, ans:'q1ans'},
  sol_q2: {prefix:'q2', count:8, ans:'q2ans'},
  sol_q3: {prefix:'q3', count:8, ans:'q3ans'}   // 只有 3 组
}
```

同时 HTML 中只有 3 组 `panel-content` 元素（q1, q2, q3），没有第 4 组的 DOM。

**修复方向：** 需要支持动态数量的解答面板。参见修复方向 A。

---

### 问题 2：解题步骤数值静态硬编码

**表现：** 步骤面板显示旧题答案（"1.0 N"、"1.41 N"），而非当前题目的正确数值。

**根因：** 步骤文字是静态 HTML，非动态计算：

```html
<div class="step-row" id="q1s2">
  <span>= <span class="color-result">1.0 N</span></span>
</div>
```

而实际的 PHYS 值已经正确（Fq = 0.4N, G = 2.0N），但未反映在步骤文本中。

**影响范围：**

| 模板 | 硬编码位置 |
|---|---|
| `electric_pendulum.html` | q1s0~q1s7, q2s0~q2s7, q3s0~q3s7, 答案框, ending 汇总 |
| `inclined_plane.html` | s1s0~s1s2, s2s0~s2s1, s3s0~s3s2, 答案框, ending 汇总 |
| `projectile.html` | s1s0~s1s2, s2s0~s2s1, s3s0~s3s2, 答案框, ending 汇总 |
| `vertical_circular.html` | 步骤无具体数值（已用公式），但 ending 汇总为占位符 |

**修复方向：** 步骤文字改为 JS 动态填充。参见修复方向 B。

---

### 问题 3：场景逻辑与问题实际条件不匹配

**表现：**
- 初始角度 θ₀ = 60° 被忽略，模板依然用 37° 和 90° 作为摆动关键角度
- 释放方式与题目描述不匹配（题目"拉到 θ₀ 位置释放"，模板先 0° 释放、再 90° 释放）

**根因：** 模板的物理场景硬编码在 `getPhysics()` 和 `SCENES` 中：

```js
// 场景固定角度
case 'swing_q2': return swing1(prog * Math.PI/2);        // 0→90° 摆动
case 'sol_q2':   return swing1(37 * PHYS.deg);            // 固定 37°
case 'pull_90':  return {theta: -prog * Math.PI/2, ...};  // 拉到 90°
case 'swing_q3': return swing2(prog);                      // -90→0° 摆动
```

而题目要求的是：拉到 θ₀=60° → 释放 → 摆到最低点 → 求 vA

**修复方向：** 场景编排应参考题目条件，动态生成角度参数。参见修复方向 C。

---

### 问题 4：模板未使用 AI 或语义理解

**表现：** 题型匹配依赖关键词打分，题目理解不深入。

**当前匹配逻辑（纯关键词）：**

```python
# electric_pendulum 的匹配关键词
tags = ['电场', '单摆', '复合场', '圆周运动', '电场力', '带电粒子']
# 匹配到 "电场" 得 10 分，"单摆" 得 10 分，汇总得 20 分
```

**局限：**
- 无法理解题目问的是什么（"求速度" vs "求拉力"）
- 无法区分相似题型（如"水平放置" vs "竖直放置"）
- 无法判断题目有几问、每问的求解目标

**修复方向：** 可选引入 LLM 做结构化理解。参见修复方向 D。

---

### 问题 5：步骤面板在场景裁剪后不匹配

**表现：** `inclined_plane.html` 场景裁剪后，`Panel.stepMap` 中的 `sol_q2`、`sol_q3` 在有粗糙面时才显示，但 smooth-only 场景下隐藏正确。不过 `ending` 面板的 `PARAMS.questions` 已动态化（阶段一已修），但步骤面板的动态化尚未完成。

**状态：** 部分修复。场景裁剪（阶段一）已完成，但步骤数值动态化（阶段二）尚未开始。

## 修复方向

### 方向 A：动态解答面板

**目标：** 根据题目问数，生成对应数量的解答面板。

**方案：**

1. 从题目文本中提取问题数量（如"（1）""求："数量）
2. 后端将问题数量传入 `__INJECTED_PARAMS__`
3. 模板 JS 根据问题数量动态生成 `panel-content` DOM 和 `stepMap`
4. 每问的步骤内容从静态 HTML 改为 JS 模板数组

```js
// 模板 JS 中定义步骤内容（非 HTML）
const _STEPS = {
  q1: [
    'F电 = qE = ${q} × ${E} = ${Fq} N',
    'G = mg = ${m} × ${g} = ${G} N',
    'F合 = √(F电² + G²) = √(${Fq}² + ${G}²) = ${Fr} N',
  ],
  q2: [...],
  // 动态创建：根据 _IN._questionCount 决定显示 q1~qN
};
```

**难度：** 中。需要重构 Panel 控制器的 DOM 生成逻辑。

---

### 方向 B：步骤数值动态化

**目标：** 步骤文字中的数字由 JS 根据 PHYS 实时计算填充。

**方案：**

每个模板的步骤 HTML 从：
```html
<div class="step-row" id="q1s2">
  <span>= <span class="color-result">1.0 N</span></span>
</div>
```

改为：
```html
<div class="step-row" id="q1s2" data-template="= %result% N">
  <span>= <span class="color-result" id="q1s2v"></span></span>
</div>
```

配合 JS 填充：
```js
function updateStepValues() {
  const Q1_VALUES = { q1s2v: PHYS.Fq.toFixed(2) };
  for (const [id, val] of Object.entries(Q1_VALUES)) {
    const el = document.getElementById(id);
    if (el) el.textContent = val;
  }
}
```

调用时机：每次场景切换到对应的 sol 场景时。

**影响范围：**

| 模板 | 需改动的步骤元素数 |
|---|---|
| `electric_pendulum.html` | 24 个步骤 + 3 个答案 + 4 个 ending 项 |
| `inclined_plane.html` | 8 个步骤 + 3 个答案 + 3 个 ending 项 |
| `projectile.html` | 8 个步骤 + 3 个答案 + 3 个 ending 项 |
| `vertical_circular.html` | 无具体数值（公式推导） |

**难度：** 低-中。机械工作量，但每个模板都需要单独处理。

---

### 方向 C：参数驱动的场景编排

**目标：** 场景内容根据题目条件（初始角度、释放方式、问题目标）动态决定。

**方案：**

将模板中的硬编码角度改为参数读取：

```js
// 当前
const THETA_TARGET = 37;           // 硬编码
const THETA_INIT = Math.PI / 2;    // 硬编码 90°

// 改为
const THETA_TARGET = _IN.theta_target ?? 37;
const THETA_INIT = _IN.theta0_deg !== undefined
  ? _IN.theta0_deg * PHYS.deg
  : Math.PI / 2;  // 默认 90°
```

场景函数参数化：

```js
function swingFrom(thetaStart, prog, totalAngle) {
  const t = thetaStart + prog * totalAngle;
  // ...物理计算...
}
```

这样 `getPhysics()` 就能根据题目条件动态生成不同起始角度和摆动范围的物理数据。

**涉及模板：** 主要是 `electric_pendulum.html`（其他模板角度依赖少）。

**难度：** 中。需要重构物理引擎函数，但不涉及 DOM 改动。

---

### 方向 D：引入 LLM 做结构化理解（可选）

**目标：** 提升题型匹配精度，提取题目问数、初始条件等结构化信息。

**方案：**

在 `template_generate()` 中增加一个可选的 LLM 步骤：

```python
# 当前：纯关键词匹配
tid = engine.match_template_by_text(text)

# 可选增强：LLM 提取结构化信息
if use_llm:
    info = llm_extract_problem_info(text)
    # info = {
    #     "template": "electric_pendulum",
    #     "question_count": 4,
    #     "initial_angle_deg": 60,
    #     "goals": ["force", "velocity", "energy", "tension"],
    # }
    tid = info.get("template", tid)
    mapped.update(info.get("params", {}))
    mapped["_question_count"] = info.get("question_count", 3)
    mapped["_goals"] = info.get("goals", [])
```

LLM 提取的目标字段：

| 字段 | 用途 | 对应问题 |
|---|---|---|
| `template` | 题型匹配 | 提升匹配准确率 |
| `question_count` | 生成对应数量的面板 | 问题 1 |
| `initial_angle_deg` | 场景初始角度 | 问题 3 |
| `goals` | 场景编排依据 | 问题 3 |
| `params` (补充提取) | 正则提取不到的参数 | 问题 2 的提取基础 |

**难度：** 取决于 LLM 服务可用性和响应速度。作为可选增强，不影响纯规则模式。
