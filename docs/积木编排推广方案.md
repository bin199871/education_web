# 积木编排系统推广方案 — 注册 16 种题型全覆盖

## 现状

当前只有 `electric_pendulum` 实现了完整的积木编排系统。其余 15 种题型处于不同阶段，架构不统一。

## 目标

将积木编排架构标准化，推广到全部 16 种注册题型，使每种题型都能：

1. 根据题目**问题目标**动态编排场景序列
2. Canvas 渲染基于 **layers 声明式**系统
3. 步骤面板**动态生成**并计算数值
4. 支持**参数注入**

---

## 一、标准化积木架构

### 1.1 通用积木库 `_COMMON`

所有模板共享以下积木，定义在 `electric_pendulum` 当前已实现的基础上，提取为可复用的模式：

```js
const _COMMON = {
  intro: {
    title: '题目介绍', dur: 0.06, goals: [],
    layers: ['bg', 'env', 'object', 'title', 'known'],
    physics: (p) => ({theta:0, x:0, v:0, ...})  // 初始静止
  },
  ending: {
    title: '答案汇总', dur: 0.08, goals: [],
    layers: ['bg', 'ending']
  }
};
```

### 1.2 积木字段规范

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `id` | string | auto | 唯一标识，与 key 同名 |
| `title` | string | ✅ | Canvas 标题 |
| `dur` | number | ✅ | 相对时长（所有积木的和归一化到 0-1） |
| `group` | string | 推荐 | `common` / 模板名 |
| `goals` | string[] | ✅ | 解决的问题目标列表 |
| `layers` | string[] | ✅ | 需要渲染的图层 |
| `physics` | function | 推荐 | `(prog) => state` 物理计算函数 |
| `transIn` | string | 可选 | 过渡文字 |
| `steps` | object | 可选 | `{title, items[], answer}` 步骤模板 |
| `answerItems` | string[] | 可选 | 答案汇总条目 |
| `slowMo` | number | 可选 | 慢放倍率（0-1，默认不用） |

### 1.3 标准图层清单

```js
// 所有模板共用的渲染图层
const LAYER_NAMES = {
  bg:      '深色渐变背景',
  title:   'Canvas 场景标题',
  env:     '物理环境（支架/斜面/轨道/导轨等）',
  object:  '运动物体（球/滑块/小车/粒子等）',
  forces:  '受力箭头',
  v_arrow: '速度箭头',
  energy:  '能量条形图',
  field:   '场线（电场/磁场线）',
  labels:  '几何标注（高度/距离/角度）',
  panel:   '解题步骤面板',
  ending:  '答案汇总面板',
  known:   '已知量面板',
}
```

### 1.4 标准编排引擎

```js
function composeScenes(goals, templateBricks) {
  const has = (g) => goals.includes(g);
  const list = [];

  // 通用开头
  list.push('intro');

  // 特定题型的场景组合逻辑（由每个模板的 compose 函数定义）
  const custom = templateCompose(goals, has);

  // 通用结尾
  list.push('ending');

  return allocateTime(list);
}
```

每个模板提供一个 `templateCompose()` 函数，将 goals 映射为该题型特有的积木序列。

## 五、风险与注意事项

### 5.1 模板文件体积

当前 `electric_pendulum.html` 约 48KB（积木编排版）。如果加入 `components.js` 的公共渲染函数引用，可以缩小到 20-30KB。其他模板预期类似。

### 5.2 目标检测精度

`GOAL_PATTERNS` 的正则匹配可能误判/漏判。建议：
- 第一阶段用正则（当前方案）
- 第二阶段引入 LLM 辅助（在 server.py 中添加可选 LLM 路径）

### 5.3 物理数值验证

每个模板的物理计算（`swing1`, `swing2` 等函数）需要经过多组参数验证，确保数值正确。

### 5.4 步骤面板的模板引擎

当前的 `${eval}` 方案简单但有安全风险（注入攻击）。由于模板是本地生成、本地播放，风险可控。生产环境应替换为安全的模板引擎。
